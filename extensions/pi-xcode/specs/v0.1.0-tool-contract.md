# pi-xcode v0.1.0 tool contract

## Scope

- iOS only
- Split tools by domain
- One setup command
- Human-first rendered output (no raw JSON in UI)

## Tools

### xcode_project

Actions:
- inspect
- list_schemes
- list_targets
- resolve_uitest_scheme
- doctor

Inputs:
- projectPath? (.xcodeproj)
- workspacePath? (.xcworkspace)
- scheme?

### xcode_build

Actions:
- build
- build_install
- build_run
- clean

Defaults:
- sensible auto-boot for simulator actions
- terminate app before reinstall/relaunch when bundle id is known

Inputs:
- scheme (required)
- projectPath? / workspacePath?
- configuration? (Debug/Release, default Debug)
- simulatorName? (required for build_install/build_run)
- bundleId? (optional, inferred when possible)
- args? (launch args)

### xcode_simulator

Actions:
- list
- status
- boot
- shutdown
- erase
- install
- launch
- terminate
- uninstall
- screenshot

Inputs:
- simulatorName? (except list)
- appPath? (install)
- bundleId? (launch/terminate/uninstall)
- args? (launch)

Screenshot behavior:
- always include saved path
- include image block in tool content

### xcode_ui

Actions:
- describe_ui
- tap
- type
- clear_text
- swipe
- scroll
- wait_for
- assert
- query_controls
- query_text
- screenshot
- chain_actions

Runner contract:
- xcode_ui executes a harness command
- payload sent as JSON on stdin
- harness responds with JSON on stdout

Inputs:
- simulatorName (required)
- runnerCommand? (default: `bash tools/ui-automation-runner.sh` if present)
- action
- params? (action-specific)
- chain_actions supports steps[]

chain_actions behavior:
- stop at first failure
- show passed steps + failing step
- checkpoint-style during assertions supported by payload model

## Rendering

- collapsed: compact summary line
- expanded: human-readable details section
- progress updates for long operations via partial updates

## Command

### /xcode:setup

Checks:
- Xcode CLI tools available
- simulator availability
- project/workspace presence
- optional harness script presence
